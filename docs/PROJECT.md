# BossSay 项目全景文档

> 从产品、用户、技术、竞品四个维度全面解析这个项目。

---

## 一、项目起源

### 1.1 问题背景

Boss直聘是中国最大的蓝领+白领招聘平台之一，日活用户超千万。求职者的核心痛点是：

- **打招呼消息千篇一律** — 大部分求职者发的是"你好，我对这个岗位很感兴趣"，HR 每天收到几百条这样的消息，根本不会回复。
- **手动写个性化消息太耗时** — 每个岗位都写不同的消息，一个求职者可能要投几十上百个岗位，根本做不到。
- **不知道怎么写才有效** — 求职者不知道 HR 看重什么，不知道怎么在 100 字内展示自己的匹配度。

### 1.2 解决方案

BossSay 是一个 Chrome/Edge 浏览器扩展，核心功能是：

1. **扫描岗位信息** — 从 Boss直聘页面自动提取职位、公司、薪资、地点等信息
2. **结合简历生成消息** — 用 AI 将岗位要求和求职者简历匹配，生成个性化打招呼消息
3. **一键填入** — 生成的消息直接填入 Boss直聘的聊天输入框

### 1.3 项目演进

| 版本 | 时间 | 里程碑 |
|------|------|--------|
| v1.0 | 2024 初 | 基础功能：手动输入 JD + AI 生成消息 |
| v2.0 | 2024 中 | PDF 简历上传、多风格、新海诚蓝色 UI |
| v2.1 | 2025 | 扫描件 OCR、求职信息字段、HR 视角 prompt 优化 |
| v3.0 | 2025 中 | 重构：搜索页卡片提取、避开 CSS 反爬、service worker API 代理 |

## 二、产品视角

### 2.1 目标用户

| 用户画像 | 特征 | 核心需求 |
|----------|------|----------|
| 应届生 | 无工作经验，投递量大 | 快速生成大量个性化消息 |
| 实习生 | 时间紧迫，需要快速找到实习 | 到岗时间、实习时长要突出 |
| 跳槽者 | 有经验，目标明确 | 突出技能匹配度和项目经历 |

### 2.2 核心用户旅程

```
用户打开 Boss直聘搜索页
        ↓
点击 BossSay 按钮打开弹窗
        ↓
点击「扫描当前页面岗位」
  → 自动提取：职位、公司、薪资、地点、经验、学历
        ↓
（可选）从详情页复制 JD 粘贴到输入框
        ↓
选择消息风格（专业/热情/幽默/简洁）
        ↓
点击「AI 生成打招呼消息」
  → AI 生成三段式消息：
    1. 能力匹配（用真实简历匹配 JD）
    2. 到岗信息（到岗时间、实习时长、转正意愿）
    3. 收尾提问（跟岗位相关的问题）
        ↓
点击「填入输入框」或「复制消息」
        ↓
用户在 Boss直聘聊天窗口发送
```

### 2.3 产品差异化

| 特点 | BossSay | 普通自动打招呼 |
|------|---------|---------------|
| 消息质量 | AI 生成，结合简历和 JD，个性化 | 固定模板，千篇一律 |
| 安全性 | 不自动发送，用户审核后手动发 | 自动发送，有封号风险 |
| 反爬对抗 | 从搜索页卡片提取，不触发安全验证 | 直接爬详情页，容易被检测 |
| 简历管理 | PDF 上传 + AI 解析 + 手动编辑 | 无简历管理 |

### 2.4 功能矩阵

| 功能 | 优先级 | 状态 | 说明 |
|------|--------|------|------|
| 岗位扫描 | P0 | ✅ | 搜索页卡片提取 |
| AI 消息生成 | P0 | ✅ | 四风格、HR 视角 prompt |
| 一键填入 | P0 | ✅ | 填入聊天输入框 |
| PDF 简历上传 | P1 | ✅ | 文字版 + 扫描件 OCR |
| 备份/恢复 | P2 | ✅ | 导出/导入 JSON |
| 历史记录 | P2 | ✅ | 保存生成过的消息 |
| 批量投递 | P3 | ❌ | 参考 BOSSING 的实现 |
| JD 智能提取 | P1 | ⚠️ | Boss直聘 CSS 反爬太强，需手动复制 |

## 三、用户视角

### 3.1 用户痛点与解决方案

| 痛点 | BossSay 的解决方案 |
|------|-------------------|
| "我不知道怎么写打招呼消息" | AI 自动生成，不需要写作能力 |
| "每个岗位都写不同的消息太累了" | 自动匹配简历和 JD，一键生成 |
| "我发的消息 HR 都不回" | 从 HR 视角设计 prompt，提高回复率 |
| "我不知道哪些技能要突出" | AI 分析 JD 要求，自动匹配简历中的相关技能 |
| "我的简历信息记不全" | PDF 上传自动提取，一次填入永久使用 |

### 3.2 使用场景

**场景 1：应届生批量投递**
- 小明是计算机专业应届生，要在 Boss直聘投 50 个前端岗位
- 打开搜索页，逐个扫描岗位，AI 自动生成匹配每个岗位的消息
- 每条消息都提到"React 项目经验"、"可实习 6 个月"、"希望转正"
- 10 分钟完成 50 个岗位的个性化消息

**场景 2：跳槽者精准投递**
- 小红有 3 年运营经验，想跳到大厂
- 上传 PDF 简历，AI 自动提取经历和技能
- 扫描岗位后，AI 结合简历生成"我做过 XX 运营，DAU 提升 30%，和你们要的增长运营很匹配"
- 消息有数据支撑，HR 愿意回复

**场景 3：实习生快速找岗**
- 小李大三，想找暑期实习
- 填写"到岗时间：7月1日后"、"可实习 4 个月"、"希望转正"
- AI 在每条消息中自然带出这些信息，HR 一眼看到可用时间

### 3.3 用户反馈驱动的迭代

| 用户反馈 | 对应改进 |
|----------|---------|
| "AI 编造了我的经历" | prompt 加铁律：只能用简历中已有的信息 |
| "消息太长了 HR 不看" | 限制 80-150 字，简洁明了风格 |
| "加载圆圈和文字一起转" | 修复 loading 动画 bug |
| "填了资料但生成时没用到" | 修复 profile 数据映射 bug |
| "Edge 浏览器用不了" | API 调用改为 service worker 代理 |
| "扫描不到 JD" | 改为从搜索页卡片提取 + 手动粘贴 JD |

### 4.2 数据流

```
[搜索页] content.js 提取卡片数据
    ↓ sendMessage
[Popup] 显示岗位信息，用户点生成
    ↓ chrome.storage.local.get
[Popup] 读取 profile + apiConfig
    ↓ chrome.runtime.sendMessage
[Service Worker] AI_CHAT_COMPLETIONS
    ↓ fetch (with cookies, no CORS)
[AI API] 返回生成的消息
    ↓ sendResponse
[Popup] 显示消息，用户编辑/填入
    ↓ chrome.tabs.sendMessage
[Content Script] 填入聊天输入框
```

### 4.4 关键技术决策

| 决策 | 原因 | 备选方案 |
|------|------|----------|
| 从搜索页卡片提取，不爬详情页 | Boss直聘详情页 CSS 反爬太强 | 拦截 API（会触发安全验证） |
| Service worker 代理 API 调用 | 避免 CORS 问题 | popup 直接 fetch（会失败） |
| pdf.js 提取 PDF 文字 | 支持 CID 字体和中文 | 简易正则（中文乱码） |
| AI 视觉识别扫描件 PDF | 不需要额外 OCR 库 | Tesseract.js（30MB） |
| 不自动发送消息 | 避免封号风险 | 自动发送（高风险） |
| `<all_urls>` host permission | 用户可能用任意 API 地址 | 只列特定域名（不灵活） |

### 4.5 技术栈

| 层级 | 技术 |
|------|------|
| 浏览器扩展 | Chrome Extension Manifest V3 |
| 弹窗界面 | 原生 HTML/CSS/JS（无框架） |
| 内容脚本 | 原生 JS，DOM 操作 |
| 后台服务 | Service Worker |
| PDF 处理 | Mozilla pdf.js |
| AI 调用 | OpenAI 兼容 API（fetch） |
| 存储 | chrome.storage.local |

### 5.2 竞品详细分析

#### BOSSING (github.com/alexH-maker/BOSSING)

**功能：**
- 搜索页批量投递（自动点击"立即沟通"）
- 关键词筛选、HR 活跃度过滤、猎头排除
- 自定义问候语 + 图片简历批量发送
- 投递记录 CSV 导出
- 三档速度控制

**技术亮点：**
- 在搜索页工作，不进详情页，避开 CSS 反爬
- 用 `card.click()` 模拟点击，`chatBtn.click()` 自动沟通
- 用 `document.execCommand('insertText')` 填入消息（兼容 React 框架）
- 侧边栏（sidePanel）UI，不干扰页面

**不足：**
- 消息是固定模板，不结合简历和 JD
- 批量操作有封号风险
- 没有 AI 能力

#### Boss直聘助手（油猴脚本）

**功能：**
- 自动打招呼（按条件筛选后批量发送）
- 数据统计（投递数、回复率）
- 页面美化

**技术亮点：**
- 油猴脚本轻量，不需要安装扩展
- 直接在页面 JS 上下文运行，无 CORS 问题

**不足：**
- 消息模板固定
- 油猴脚本安全性较低
- 更新维护不稳定

## 六、面试准备（八股文）

### 6.1 产品角度

**Q: 你的项目解决了什么问题？**
> Boss直聘求职者发的打招呼消息千篇一律，HR 不回复。BossSay 用 AI 结合简历和 JD 生成个性化消息，提高 HR 回复率。

**Q: 你的目标用户是谁？**
> 主要是应届生和实习生，投递量大、经验少、不知道怎么写有效消息。也适合有经验的跳槽者，突出技能匹配度。

**Q: 你怎么验证产品需求的？**
> 1. Boss直聘日活千万，求职是刚需
> 2. 竞品（BOSSING、油猴脚本）有用户但消息质量差
> 3. 用户反馈驱动迭代（AI 编造经历→加铁律、loading bug→修复）

**Q: 你的产品和竞品有什么区别？**
> 核心区别是消息质量。竞品用固定模板，所有消息一样。BossSay 用 AI 结合简历和 JD 生成个性化消息，每条都不同。同时我们不自动发送，避免封号风险。

**Q: 你怎么考虑商业化？**
> 短期开源免费，积累用户。中期可以上 Chrome Web Store，高级功能收费。长期可以做成 SaaS，但需要服务器成本。

### 6.2 技术角度

**Q: Chrome 扩展的架构是什么？**
> Manifest V3 架构，三个核心部分：
> 1. Popup — 弹窗界面，用户交互入口
> 2. Content Script — 注入到 Boss直聘页面，提取数据和填入消息
> 3. Service Worker — 后台服务，处理存储和 API 调用
>
> 通信方式：`chrome.runtime.sendMessage` 在三者之间传递消息。

**Q: 你遇到的最大技术挑战是什么？**
> Boss直聘的 CSS 反爬。详情页的 JD 内容用自定义字体+CSS 混淆，DOM 里只有 CSS 代码，没有真实文字。我们尝试了：
> 1. 直接 DOM 提取 → 拿到 CSS 代码
> 2. 拦截 API 响应 → API 在 content script 注入前就完成了
> 3. 主动调 API → 触发安全验证
> 4. 注入 CSS 解锁选中 → 文字本身就是 CSS，选中也没用
>
> 最终方案：参考竞品 BOSSING，从搜索页卡片提取（搜索页不混淆），JD 需用户手动粘贴。

**Q: 为什么用 Service Worker 代理 API 调用？**
> Chrome 扩展的 popup 直接 fetch 外部 API 会遇到 CORS 限制。Service Worker 的 fetch 不受页面 CORS 约束，因为它是扩展的后台上下文，有 `host_permissions` 授权。所以所有外部 API 调用都通过 service worker 代理。

**Q: PDF 提取怎么实现的？**
> 三级策略：
> 1. **pdf.js** — Mozilla 的 PDF 解析库，支持 CID 字体和中文
> 2. **渲染为图片** — pdf.js 渲染每页为 canvas，转 base64 JPEG
> 3. **AI OCR** — 图片发给视觉模型（如 GPT-4o）做文字识别
>
> 对于扫描件 PDF（图片），走 2→3 路径。对于文字版 PDF，走 1 路径。

**Q: 你的 prompt 工程做了什么？**
> 从 HR 视角设计：
> 1. **铁律** — 禁止 AI 编造简历中没有的经历和数据
> 2. **三段式** — 能力匹配 → 到岗信息 → 收尾提问
> 3. **四风格** — 专业/热情/幽默/简洁，每种风格都包含 JD 匹配和到岗信息
> 4. **反面教材** — 列出绝对不要写的表达（如"我对贵公司很感兴趣"）

**Q: 数据存储怎么设计的？**
> 用 `chrome.storage.local`，所有数据存在浏览器本地，不上传服务器。存储结构：
> - `bossSay_profile` — 求职者资料（简历、经历、技能、到岗时间等）
> - `bossSay_apiConfig` — AI 配置（API 地址、Key、模型名）
> - `bossSay_history` — 生成历史
>
> 通过 service worker 的消息处理器做 CRUD，popup 和 options 页面都通过 sendMessage 访问。

**Q: 你怎么处理 SPA 页面的 URL 变化？**
> Boss直聘是 SPA（单页应用），URL 变化不会触发页面刷新。用 `MutationObserver` 监听 DOM 变化，检测 `location.href` 是否改变。如果变了，清除旧数据，重新注入按钮。

**Q: 如果让你继续优化，你会做什么？**
> 1. **批量扫描** — 参考 BOSSING，搜索页一次扫描所有卡片
> 2. **HR 活跃度过滤** — 显示 HR 在线状态，优先投递活跃 HR
> 3. **投递记录导出** — CSV 导出，方便复盘
> 4. **JD 智能提取** — 如果 Boss直聘开放 API 或降低反爬强度
> 5. **消息效果追踪** — 统计哪些消息风格回复率高
